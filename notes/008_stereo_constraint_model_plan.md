# Stereo Constraint Model Plan

Branch: `stereo-constraint-model`

## Goal

Replace ad hoc bond-stereo carrier heuristics with a small native constraint
model for stereo precomputation.

The model should not be a Z3 runtime dependency. Z3 remains an exploratory
oracle for discovering and checking the constraint language. Production code
should use a compact native representation that can answer online completion
queries cheaply.

## Long Game

The long-term target is to make bond-stereo handling an explicit constraint
problem instead of a collection of walker-local heuristics.

At startup, the molecule should decompose into independent stereo components.
Each component should carry a compact native model over carrier choices,
direction-token orientation, shared carrier edges, and eventually traversal
facts such as tree edges, ring closures, and branch context. During online
enumeration or decoding, the walker should not repair a completed SMILES and
should not guess from scattered special cases. It should emit facts as the
walk proceeds and ask the model whether the partial assignment still has a
completion under the selected contract.

The selected contract must remain explicit:

- semantic SMILES validity and stereo preservation
- RDKit local writer cleanup behavior
- full RDKit writer parity, including traversal and ring-closure policy

This separation is the point. RDKit-specific behavior can then be mirrored
where the public API requires RDKit writer support, while still remaining
visibly separate from principled SMILES/chemistry semantics. If Grimace later
exposes a broader semantic-support mode, it should be a distinct mode, not an
accidental byproduct of RDKit parity logic.

The first milestone is intentionally modest: create the model boundary, build
it at runtime, and prove current walker choices are accepted. Only after that
should suspicious heuristics be replaced one rule at a time.

## Correctness Layers

Keep these layers explicit in names, fixtures, and tests:

- `Semantic`: SMILES carrier choices that are chemically/language valid and
  parse back to the intended graph and stereo assignment.
- `RdkitLocalWriter`: RDKit local writer cleanup policies, such as avoiding
  directional carriers on both sides of a non-stereo double bond.
- `RdkitTraversalWriter`: RDKit traversal-, ring-closure-, branch-, and
  serializer-policy constraints that are not pure local chemistry rules.

The current public API is RDKit writer parity, so production pruning must
eventually apply RDKit writer layers. The semantic layer remains useful as a
principled internal boundary and as evidence for whether Grimace can support a
broader future mode.

## Target Model

The native model should be a small CSP over stereo-carrier facts:

- Variables:
  - carrier-edge selection per stereo endpoint
  - token orientation/phase for selected carrier edges
  - emitted carrier facts observed during traversal
  - traversal-role facts where needed, such as tree edge vs ring closure
- Domains:
  - one or more eligible carrier edges per endpoint
  - compact integer/bitset domains, not string-driven state
- Constraints:
  - endpoint carrier validity
  - same-side consistency for two candidate carriers
  - E/Z across-double-bond parity
  - shared-carrier coupling across endpoints
  - non-stereo-double hazard exclusion
  - RDKit traversal/ring writer exclusions once derived cleanly
- Components:
  - independent stereo components should remain independent
  - coupled stereo bonds should live in the same component

The first native implementation can use explicit small allowed-assignment
bitsets per component. That is acceptable if it is generated by general
constraints, not handwritten per witness. If component domains grow too large,
replace enumeration with clause propagation later.

## Integration Phases

### Phase 1: Behavior-Preserving Scaffold

- Add `StereoConstraintModel` and related typed structures in
  `rust/src/bond_stereo_constraints.rs`.
- Build the model next to the existing `StereoSideInfo` preprocessing.
- Initially encode constraints already implied by current preprocessing.
- Store the model in `StereoWalkerRuntimeData` as a no-op field.
- Do not change enumeration or decoder support.

Expected commit:

- `Add native stereo constraint model scaffold`

### Phase 2: Model Fixtures and Tests

- Add pinned model fixtures for witnesses found by exploration.
- Do not require Z3 in CI.
- Assert counts and bit patterns that came from the exploratory scripts:
  - minimal non-stereo-double hazard witness:
    - semantic assignments: 4
    - RDKit local writer assignments: 3
  - reduced porphyrin witness:
    - semantic assignments: 16
    - RDKit observed writer assignments: 12
- Add Rust unit tests for model mechanics:
  - component independence
  - allowed-assignment projection
  - incremental fact application
  - empty/no-stereo model behavior

Expected commit:

- `Add stereo constraint model fixtures`

### Phase 3: Runtime Validation, Still No Pruning

- Add debug/test-only checks that current walker carrier selections are
  accepted by the model.
- Check both terminal completions and intermediate emitted carrier facts where
  feasible.
- Keep these checks internal; do not expose model concepts in public API.

Expected commit:

- `Validate current stereo choices against model`

### Phase 4: Replace Local RDKit Cleanup Heuristic

- Implement the non-stereo-double hazard as a model constraint.
- Use the model to reject impossible local carrier assignments.
- Keep traversal/cycle coupling heuristics unchanged until the rule is derived
  from traversal state instead of bit-pattern observation.

Expected commit:

- `Use model for local bond-stereo hazard constraints`

### Phase 5: Traversal Writer Policy

- Derive RDKit traversal-policy facts from online walker state:
  - atom output order
  - bond output order
  - ring opening/closure role
  - branch context where needed
- Add `RdkitTraversalWriter` constraints only after the rule is stated in
  graph/traversal terms.
- Reduced porphyrin remains the first target: RDKit has 12 observed assignment
  patterns while the semantic-local model has 16.

Expected commit:

- `Model RDKit traversal stereo policy`

## Test Boundaries

Keep the test layers separate:

- Native model unit tests:
  - no RDKit
  - no Z3
  - assert CSP mechanics and component decomposition
- Pinned RDKit fixture tests:
  - version-keyed RDKit expectations
  - assert known witness counts and bit patterns
  - no random sampling in CI
- Runtime parity tests:
  - prove enumeration and decoder behavior match pinned RDKit support
  - should remain unchanged during Phase 1 through Phase 3
- Exploratory scripts:
  - may use Z3 and RDKit sampling
  - live under `tmp/exploration/`
  - generate evidence, not CI requirements

## Proceeding Strategy

Every change should be either behavior-preserving or justified by a pinned
witness. The model exists to make the rules explicit; it should not become a
new place to hide untested special cases.

Proceed in this order:

1. Commit planning and terminology before implementation.
2. Add the native model scaffold without pruning runtime behavior.
3. Add fast unit tests for model shape and mechanics.
4. Build the model in runtime as a no-op field.
5. Add debug/test-only validation that current walker choices are accepted.
6. Add pinned model fixtures from the exploration witnesses.
7. Replace only the local non-stereo-double hazard first.
8. Re-run full parity tests and compare known witnesses.
9. Tackle traversal/cycle policy only after its rule is stated in graph and
   traversal terms.

Do not make a runtime pruning change until the model can explain that case in
a small test. If a rule cannot be stated as a constraint with clear inputs and
an expected projection, leave it in `tmp/exploration/` until it can.

## Immediate First Step

Add a behavior-preserving `StereoConstraintModel` scaffold with enough shape to
represent:

- component ids
- side ids by component
- side choice domains
- per-layer allowed-assignment storage
- a `has_completion(component, facts, layer)` query

The first implementation may return broad acceptance for all current runtime
facts. The important part is to land the stable API and tests for the model
shape before replacing heuristics.
